How This Site Was Built

This entire landing page was created using Claude Code — Anthropic's AI-powered command-line tool. Here's how it came together.

Tool Overview

Claude Code is an agentic coding tool that lives in your terminal. You give it a task in natural language, and it plans, writes, edits, and executes code — using your real filesystem, running real shell commands, and iterating based on results.

It doesn't generate code in a chat bubble for you to copy-paste. It directly creates files, edits them in place, runs git commands, and deploys — acting as a collaborator that happens to operate at the speed of an LLM.

For this project, Claude Code handled everything: file creation, CSS styling, JavaScript logic, git operations, and GitHub Pages deployment. The human's role was to provide the plan and approve actions along the way.

The Prompting Approach

Getting a polished result from an AI coding tool isn't about writing one perfect prompt — it's about structuring the conversation. Here's the general strategy that worked:

1. Start with the big picture

The first prompt described the goal at a high level: "Create a polished demo landing page to showcase Claude Code's capabilities." This let the AI ask clarifying questions rather than making assumptions.

2. Make decisions upfront

Before writing any code, key decisions were locked in: tech stack (vanilla HTML/CSS/JS), visual style (dark modern), sections to include, and hosting (GitHub Pages). This prevented mid-stream pivots that waste time.

3. Provide a structured plan

Rather than vague instructions, the implementation was laid out as a numbered list of concrete steps — create repo, build each file, deploy. This gives the AI clear checkpoints and prevents it from trying to do everything at once.

4. Let the AI handle details

The plan specified what each section should contain but didn't dictate exact HTML or pixel-perfect CSS. Giving the AI room to make implementation decisions (color values, spacing, SVG icons) plays to its strengths.

5. Iterate if needed

After the initial build, you can review and refine — "make the gradient more subtle," "add more spacing to the pricing cards," etc. Each iteration is fast because the AI has full context.

Tech Stack Decisions

Every choice was made to optimize for speed-to-live-site. Here's the reasoning:

HTML/CSS/JS vs. React/Frameworks

Pros

Zero build step. No npm install, no bundler config, no waiting for compilation. Files work the moment they exist. GitHub Pages serves them directly.

Cons

No component reuse — the nav is duplicated across pages. No hot module replacement during development. Manual responsive styling.

Why chosen: For a two-page demo site, the overhead of a framework (setup, dependencies, build pipeline) would take longer than the time it saves. Vanilla HTML/CSS/JS is the fastest path from zero to deployed.

Dark Modern vs. Light/Colorful

Pros

High visual impact. Gradient accents pop against dark backgrounds. Matches the aesthetic of tools developers already respect (Linear, Vercel, Raycast). Feels premium.

Cons

Accessibility requires more care — contrast ratios need verification. Some users prefer light themes. Dark-on-dark borders can look muddy if not tuned.

Why chosen: This page is a portfolio piece / speed test, not a production app. The dark modern aesthetic creates the strongest first impression for a developer audience and is what people expect from a "cutting-edge" tool demo.

GitHub Pages vs. Other Hosting

Pros

Free. Zero configuration — just push to main and enable Pages. Tightly integrated with the repo. Custom domain support if needed later.

Cons

Static files only (no server-side logic, no API routes). Build pipeline options are limited. CDN is decent but not as fast as Vercel/Cloudflare.

Why chosen: Since this is a static site with no backend, GitHub Pages is the fastest path from git push to live URL. Literally one API call to enable it.

Why No Extra Tooling

This project was built entirely with Claude Code's built-in capabilities:

  • File Write/Edit — to create and modify HTML, CSS, and JS files
  • Bash — to run git and gh CLI commands for repo creation and deployment

No MCP servers, no external skill packs, no browser automation, no design tools. Why?

  • Setup time kills speed. Configuring an MCP server or installing a skill pack takes minutes that defeat the purpose of a speed demo.
  • Vanilla HTML/CSS/JS doesn't need build tooling. There's nothing to compile, transpile, or bundle. The files work as-is.
  • The constraint proves the point. If you can build a polished, deployed landing page with just file editing and shell commands, the tool is powerful enough for serious work.

The Timeline

From the first prompt to a live, deployed site on GitHub Pages:

One conversation. Minutes, not hours.

The process looked like this:

  1. Plan the approach and make key decisions (tech, style, structure)
  2. Create the GitHub repository
  3. Build all four files (index.html, styles.css, script.js, how-it-was-made.html)
  4. Commit, push, and enable GitHub Pages
  5. Verify the live site

The AI parallelized where possible — creating independent files simultaneously rather than sequentially. The human provided the plan and approved each step. The result is what you're looking at right now.
